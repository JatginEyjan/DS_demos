## 一、总体评价

v0.1 的代码实现已覆盖核心循环的绝大部分设计：

- 场景流转（menu → planning → execution → settlement → boss → gameover/victory）
- 卡牌三大类（路程 / 镶嵌 / 角色）+ 品质分级 + 镶嵌品质限制
- 时间轴系统（耐力预算、日夜 50% 分界、跨昼夜乘数）
- 魂系六维属性 + 武器补正 S/A/B/C/D + 强化 +0~+8
- 灵魂双用（升级 & 消费）
- 一命容错死亡 + 灵魂掉落回收
- Boss 三阶段（策略 + 应对选择）
- 篝火、铁匠、商人、防火女、巡礼者
- 镶嵌系统、NPC 冷却、事件卡消耗
- 不死系相邻 Combo（×1.5）
- 跨天预布局
- 结算三选一奖励
- 日志系统 + localStorage 存档
- 暗黑奇幻风格 UI + 响应式布局

以下按优先级分三级列出待修复 / 补充的条目。

## 二、P0——逻辑缺陷 / Bug（建议优先修复）

### #1 装备武器不检查属性需求

**文件**：`src/systems/InventorySystem.js` → `equip()`

**现状**：`equip()` 直接赋值 `hero.weaponId = cardId`，没有校验 hero 属性是否满足武器的 `requirements`。例如大剑要求力量 18 / 敏捷 10，但即使属性不达标也能装备。

**影响**：破坏了 Build 路线限制——玩家可以在 Day 1 就装备高需求武器，"属性需求"形同虚设。

**修复方案**：在 `equip()` 中遍历 `template.requirements`，逐项校验 `hero.attributes[attr] >= required`，不满足时返回失败原因。同时在 UI 层（`InventoryPanel.js`）给不满足需求的装备按钮加上 `disabled`。

---

### #2 背包容量没有强制执行

**文件**：涉及所有向 `state.inventory.push()` 写入的位置

- `CombatSystem.applyLoot()`
- `CombatSystem.resolveRoomSlot()`（巡礼者给卡）
- `GameSession.chooseReward()`
- `CombatSystem.buyMerchantOffer()`

**现状**：`hero.maxInventorySize` 有计算（10 + 力量 / 2），UI 也展示了 `容量 x/y`，但获取卡牌时**从未检查是否超出上限**，可以无限囤积。

**影响**：力量属性的"背包扩容"系统效果完全失效，资源管理维度缺失。

**修复方案**：封装一个 `InventorySystem.addCard(card)` 方法统一做容量检查。溢出时：
- 战斗掉落 → 自动分解为灵魂（并记日志）
- 商人购买 / 奖励选择 → 阻止操作并提示"背包已满"

---

### #3 执行阶段节奏过快

**文件**：`src/config/constants.js`

**现状**：

```
EXECUTION_MS_PER_HOUR = 450
```

按此配置，3h 房间只需 **1.35 秒**实时通过，5h 房间 **2.25 秒**。整个 Day 1（假设 7h 卡牌）约 3 秒就跑完。

**影响**：玩家几乎来不及阅读日志、做微决策（使用治疗瓶 / 消耗品），"自走阶段"退化为瞬间跳转。GDD 建议一天的执行阶段应有 30~60 秒。

**修复方案**：将 `EXECUTION_MS_PER_HOUR` 调至 `2500~3500` 范围（7h 的一天约 17~24 秒执行时间）。可提供一个"加速"按钮让玩家手动倍速。

## 三、P1——设计文档明确要求但代码未实现

### #4 `InventorySystem.findCardEverywhere` 硬编码天数

**文件**：`src/systems/InventorySystem.js` 第 16 行

**现状**：

```javascript
for (const day of [1, 2, 3]) {
```

**影响**：如果切换到 6 天 / 8 天路线，Day 4+ 的卡牌将无法被查找到，导致装备引用丢失、预布局失效等连锁问题。

**修复方案**：

```javascript
for (const day of Object.keys(this.session.state.layouts).map(Number)) {
```

---

### #5 Combo 检测用 `findIndex(templateId)` 可能定位错误

**文件**：`src/systems/CombatSystem.js` → `getRoomComboMultiplier()`

**现状**：通过 `findIndex` 查找第一个匹配 `templateId` 的卡槽。如果同一 `templateId` 的卡牌出现多次（消耗版房间 + 永久版同时存在），会找到错误的位置，导致邻居判定失准。

**修复方案**：传入当前正在执行的 `slotIndex`（来自 `execution.slotIndex`）而非 `templateId` 来定位，彻底避免歧义。

---

### #6 房间探索度——有追踪但无效果

**文件**：`src/core/GameSession.js` → `incrementExploration()` / `src/systems/CombatSystem.js` → `resolveRoomSlot()`

**现状**：`incrementExploration()` 在 `state` 上记录了 `{roomTemplateId}_exploration` 计数，但 `resolveRoomSlot()` **从未读取此值**来改变房间行为。

**GDD 设计**：

| 探索度 | 效果 |
|--------|------|
| 第 1 次 | 基础怪物 + 基础掉落 |
| 第 2 次 | 解锁隐藏区域（额外战斗 + 额外掉落） |
| 第 3 次 | 触发特殊事件（一次性奖励） |
| 第 4 次+ | 效果衰减（灵魂 -30%） |

**修复方案**：在 `resolveRoomSlot()` 开头读取探索度，根据等级追加额外灵魂、额外材料、额外事件卡掉落，或施加衰减系数。

---

### #7 Boss 逼近度系统过于简化

**文件**：`src/systems/BossApproachSystem.js`

**现状**：整个系统只有 1 个方法 `advanceDay()`（+10）。Boss 战的触发纯靠天数 `day >= totalDays`。

**GDD 设计**：

| 来源 | 逼近度变化 | 当前实现 |
|------|-----------|---------|
| 每过一天 | +10~15（按路线变化） | 固定 +10 |
| 跳过精英战 | +5 | ❌ 未实现 |
| 使用紫色+卡 | +2 | ❌ 未实现 |
| 击败精英 | -3 | ❌ 未实现 |
| ≥100 时 | 强制 Boss 战，Boss +15% 强化 | ❌ 未实现 |
| 70~99 时 | 可主动挑战，额外奖励 +20% | ❌ 未实现 |

**修复方案**：
1. 在 `SettlementScene` 的"跳过精英"/"挑战精英"选项中调用逼近度增减
2. 在 `PreLayoutSystem.placeCard()` 中检测紫色+卡并 +2
3. 在 `GameSession.nextDay()` 中加入 `bossApproach >= 100` 的强制 Boss 检查
4. Boss 强制战时对 `BOSS_PHASES` 各阶段攻防 ×1.15

---

### #8 死亡灵魂掉落缺少 UI 提示

**文件**：`src/scenes/PlanningScene.js` / `src/ui/DaySchedule.js`

**现状**：`DeathSystem` 逻辑完善（掉落 + 回收），但规划阶段 UI **没有任何提示**告诉玩家灵魂掉落在哪个房间、需要安排哪张卡去回收。

**影响**：这是魂系死亡机制的核心体验（"我一定要拿回来"的执念），缺少提示等于核心体验被隐藏。

**修复方案**：当 `deathState.droppedSouls > 0` 时，在规划阶段顶部显示醒目的警告横幅：

```
⚠ 你有 {N} 灵魂掉落在「{房间名}」中。请在今天安排该房间卡以回收灵魂，否则再次死亡将永久失去。
```

## 四、P2——偏差 / 缺失内容（不影响核心运行但偏离设计）

### #9 属性软上限未实现

**文件**：`src/systems/HeroSystem.js` → `syncDerivedStats()`

**GDD 设计**：

| 属性 | 软上限 |
|------|--------|
| 生命力 | 25 |
| 耐力 | 22 |
| 力量 / 敏捷 / 智力 / 信仰 | 30 |

超过软上限后每级收益减半。

**现状**：属性增长完全线性，无上限。

**修复方案**：在 `syncDerivedStats()` 中对每项属性计算有效值时加入软上限逻辑：

```javascript
function effectiveAttr(raw, softCap) {
  if (raw <= softCap) return raw;
  return softCap + (raw - softCap) * 0.5;
}
```

### #10 骰子系统返回值过于简化

**文件**：`src/systems/DiceSystem.js`

**技术文档设计**：

```typescript
interface DiceResult {
  rawRoll: number;
  modifier: number;
  finalValue: number;
  outcome: 'critical_fail' | 'fail' | 'success' | 'critical_success';
}
```

**现状**：`roll()` 直接返回一个 `number`，调用方无法获取原始骰值、修正值、结果分类。

**修复方案**：返回结构化对象，便于 UI 展示骰子动画和结果高亮。

### #11 掉落系统是确定性的，不是概率性的

**文件**：`src/systems/CombatSystem.js` → `resolveRoomSlot()`（第 152 行）

**现状**：

```javascript
(roomData.loot || []).forEach((lootTemplateId) => this.applyLoot(lootTemplateId));
```

房间的 `loot` 数组会**必定掉落所有**列出的物品。

**MVP 卡池文档设计**：

| 来源 | 掉落概率 |
|------|---------|
| 白色怪物 | 5% |
| 蓝色怪物 | 10% |
| 精英怪 | 30% |

**修复方案**：对每个 loot 项加入概率判定：

```javascript
(roomData.loot || []).forEach((lootTemplateId) => {
  const rate = template.quality === 'rare' ? 0.25 : 0.4;
  if (Math.random() < rate) this.applyLoot(lootTemplateId);
});
```

### #13 篝火放置层不限次数

**文件**：`src/systems/PreLayoutSystem.js` → `canPlaceCard()`

**GDD 设计**：篝火卡每天限使用 1 次。

**现状**：执行阶段通过 `bonfireUsed` 标记确保只生效一次，但**规划阶段不阻止玩家放入两张篝火卡**到同一天。第二张篝火仍然会在执行时触发（回复 HP + 刷新瓶），因为 `bonfireUsed` 只在第一张篝火执行后才设为 `true`。

**修复方案**：在 `canPlaceCard()` 中检查目标天已有多少张篝火卡：

```javascript
const existingBonfires = slots.filter(s => s.card.templateId === 'time_bonfire').length;
if (template.id === 'time_bonfire' && existingBonfires >= 1) {
  return { ok: false, reason: '每天最多放置 1 张篝火卡。' };
}
```

## 五、P3——架构 / 可维护性改进（不影响当前功能）

### #14 EventBus 几乎未使用

**文件**：`src/utils/event-bus.js` / 全局

**现状**：事件总线只用于 `log:added` 一个事件。技术文档定义了丰富的事件类型（`hero:damage`、`combat:monster_killed`、`timeline:slot_enter`、`economy:souls_changed` 等），但所有系统间通信都通过直接方法调用。

**影响**：当前规模下不是问题，但随着系统扩展（稀有遭遇、成就、音效触发等），直接调用会导致耦合加重。

**建议**：在后续版本中逐步将关键节点（战斗结果、灵魂变化、阶段切换）改为事件驱动。

---

### #15 战斗公式与 GDD 不一致

**文件**：`src/systems/CombatSystem.js` → `resolveRoomSlot()`

**GDD 定义**（收益递减模型）：

```
物理伤害 = 攻击力 × (1 - 防御力 / (防御力 + 100))
```

**当前实现**（平坦减法模型）：

```javascript
const rawDamage = Math.max(6, roomData.power * incomingMultiplier - hero.defense * 0.42);
```

两种模型的曲线行为不同——当前模型在高防时减伤效果不如递减模型平滑。如果正式版需要精确数值平衡，建议切到 GDD 公式。

---

### #16 存档版本迁移未实现

**文件**：`src/utils/save.js`

**技术文档设计**：存档包含 `version` 字段，加载时执行 `migrateSave(data)` 迁移函数。

**现状**：`loadSave()` 直接 `JSON.parse`，无版本检查。`hydrateState()` 做了部分容错（补充缺失字段），但不是正式的版本迁移。

**影响**：当数据结构变更（如新增卡牌字段、修改属性命名）时，旧存档会静默出错。

**建议**：在 `loadSave()` 中加入版本判断和迁移链。

---

### #17 房间特殊效果硬编码在 CombatSystem 中

**文件**：`src/systems/CombatSystem.js`（第 109~114 行）

**现状**：古老教堂白天的特殊效果（回复 HP、临时 +1 瓶）直接用 `if (template.id === 'room_old_church')` 硬编码。

**影响**：每新增一个有特殊效果的房间卡，都需要修改 `CombatSystem`。随着卡池扩大，这个文件会越来越臃肿。

**建议**：将房间特殊效果抽象为数据驱动的 effect 配置：

```javascript
// 在 room card data 中定义
effects: [
  { trigger: 'day', type: 'heal_percent', value: 0.18 },
  { trigger: 'day', type: 'flask_temp', value: 1 }
]
```

然后在 `resolveRoomSlot()` 中统一遍历 `template.effects` 执行，不再按 ID 判断。

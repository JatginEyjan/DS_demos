## 1. 技术选型

### 1.1 推荐方案：Phaser 3 + TypeScript

| 维度 | 选型 | 理由 |
|------|------|------|
| 游戏引擎 | **Phaser 3** | 成熟的 H5 2D 游戏引擎，社区活跃，文档完善 |
| 开发语言 | **TypeScript** | 类型安全，大型项目可维护性好 |
| 构建工具 | **Vite** | 快速 HMR，原生 TS 支持 |
| UI 框架 | **DOM + CSS**（游戏外 UI） | 菜单、背包等复杂 UI 用 DOM 更高效 |
| 状态管理 | **自研状态机 / Zustand** | 游戏状态复杂，需要清晰的状态管理 |
| 音频 | **Howler.js** | 跨浏览器音频兼容性好 |
| 数据存储 | **localStorage + IndexedDB** | 本地存档，无需后端 |

### 1.2 备选方案

| 方案 | 引擎 | 适用场景 |
|------|------|---------|
| 方案 B | **PixiJS 7 + TypeScript** | 如果 Phaser 太重，PixiJS 更轻量灵活 |
| 方案 C | **Cocos Creator** | 如果未来需要原生移动端导出 |
| 方案 D | **纯 Canvas + TypeScript** | 极简方案，完全自研 |

### 1.3 开发环境

| 工具 | 版本 |
|------|------|
| Node.js | ≥ 18.x |
| TypeScript | ≥ 5.x |
| Phaser | 3.80+ |
| Vite | 6.x |
| 包管理 | pnpm |

### 2.2 架构模式

采用 **ECS-like（实体组件系统风格）** 的模块化架构：

```
Scenes（场景流转）
  ├── 调用 Systems（核心逻辑）
  │     ├── TimelineSystem（时间轴推进）
  │     ├── CombatSystem（战斗计算）
  │     ├── CardSystem（卡牌管理）
  │     └── ...
  ├── 操作 Entities（游戏实体）
  │     ├── Hero（勇者数据）
  │     ├── Monster（怪物实例）
  │     └── ...
  └── 渲染 UI Components（视觉组件）
        ├── TimelineBar
        ├── CardSlot
        └── ...
```

各 System 之间通过**事件总线（EventBus）**通信，避免直接耦合。

## 3. 核心模块分解

### 3.1 时间轴系统（TimelineSystem）

**职责**：管理每天的卡槽序列、时间推进、日夜切换。

**核心接口**：

```typescript
interface TimelineSystem {
  // 初始化一天的时间轴
  initDay(dayIndex: number, slots: CardSlot[]): void;

  // 开始执行（实时推进）
  startExecution(): void;

  // 暂停 / 恢复
  pause(): void;
  resume(): void;

  // 每帧更新
  update(delta: number): void;

  // 获取当前状态
  getCurrentSlotIndex(): number;
  getElapsedHours(): number;
  isDaytime(): boolean;
  isExecuting(): boolean;

  // 事件
  onSlotEnter: Signal<CardSlot>;    // 进入新卡槽
  onSlotExit: Signal<CardSlot>;     // 离开卡槽
  onDayNightSwitch: Signal<boolean>; // 日夜切换
  onDayEnd: Signal<void>;           // 一天结束
}
```

**时间推进逻辑**：

```
游戏实时 1 秒 = 游戏内 N 小时（可配置的时间缩放比）
MVP 建议：1 秒 = 0.5h → 一个 3h 的卡槽需要 6 秒实时通过
```

### 3.3 战斗系统（CombatSystem）

**职责**：自动战斗的伤害计算、回合管理、掉落生成。

**核心接口**：

```typescript
interface CombatSystem {
  // 开始战斗
  startCombat(hero: Hero, monsters: Monster[]): void;

  // 每帧更新（自动攻击循环）
  update(delta: number): void;

  // Boss 战策略选择
  selectBossStrategy(strategy: BossStrategy): void;
  selectBossResponse(response: BossResponse): void;

  // 事件
  onDamageDealt: Signal<DamageEvent>;
  onMonsterKilled: Signal<Monster>;
  onLootDrop: Signal<LootEvent>;
  onHeroDeath: Signal<void>;
  onCombatEnd: Signal<CombatResult>;
  onBossPhaseChange: Signal<number>;
  onBossAttackWarning: Signal<BossAttack>;
}
```

**伤害公式实现**：

```typescript
function calculateDamage(attacker: CombatStats, defender: CombatStats): number {
  const baseDamage = attacker.attack;
  const scaledDamage = baseDamage * (1 + getScalingBonus(attacker));
  const mitigated = scaledDamage * (1 - defender.defense / (defender.defense + 100));

  // 闪避判定
  if (Math.random() < defender.dodgeRate) return 0;

  // 暴击判定
  const critMultiplier = Math.random() < attacker.critRate ? 1.5 : 1.0;

  return Math.floor(mitigated * critMultiplier);
}
```

### 4.1 卡牌数据结构

interface Card {

## 6. 渲染与动画

### 6.1 场景层级

```
Layer 0: 背景层（房间背景 / 天空 / 日夜渐变）
Layer 1: 地面层（地形元素）
Layer 2: 实体层（怪物、NPC、勇者）
Layer 3: 特效层（攻击特效、伤害数字、暴击闪光）
Layer 4: UI 层（时间轴条、状态栏）
Layer 5: 弹窗层（骰子事件、战利品选择、NPC 对话）
Layer 6: 过渡层（场景切换动画）
```

### 6.2 关键动画需求

| 动画 | 实现方式 | 帧率 |
|------|---------|------|
| 勇者行走 | 精灵帧动画（4~8 帧） | 8 FPS |
| 勇者攻击 | 精灵帧动画（4~6 帧） | 12 FPS |
| 怪物受击 | 闪白 + 位移 | - |
| 伤害数字 | 浮动文字（Tween 上飘 + 淡出） | 60 FPS |
| 暴击特效 | 粒子效果 + 放大数字 | 60 FPS |
| 骰子动画 | 精灵旋转 + 停止 | 30 FPS |
| 日夜过渡 | 全屏渐变叠加色 | 60 FPS |
| 卡槽切换 | 左滑过渡 | 60 FPS |
| Boss 大招预警 | 屏幕抖动 + 红色闪光 | 60 FPS |

### 6.3 适配方案

| 维度 | 方案 |
|------|------|
| 基准分辨率 | 1280 × 720（16:9） |

## 8. 存档与持久化

### 8.1 存储策略

| 数据类型 | 存储位置 | 说明 |
|---------|---------|------|
| 局内存档 | localStorage | 当前运行中的局状态（自动保存） |
| 局外进度 | IndexedDB | 职业解锁、遗物解锁、路线解锁 |
| 设置 | localStorage | 音量、画质、语言等 |

### 8.2 自动存档时机

| 时机 | 保存内容 |
|------|---------|
| 每天规划阶段开始时 | 完整 GameState |
| 结算阶段结束时 | 完整 GameState |
| 玩家手动暂停时 | 完整 GameState |
| Boss 战开始前 | 完整 GameState |

### 8.3 存档数据格式

```typescript
interface SaveData {
  version: string;           // 存档版本号
  timestamp: number;
  gameState: GameState;      // 完整游戏状态
}

interface MetaProgress {
  version: string;
  unlockedClasses: string[];
  unlockedRelics: string[];
  unlockedRoutes: string[];
  totalRuns: number;
  totalWins: number;
  bestScores: Record<string, number>;
}
```

### 8.4 版本兼容

存档包含 `version` 字段，代码中实现迁移函数：

```typescript
function migrateSave(data: SaveData): SaveData {
  if (data.version === '0.1') {
    // v0.1 -> v0.2 的迁移逻辑
  }
  return data;
}
```

### 11.3 技术风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| 规划阶段拖放 UI 在移动端体验差 | 高 | MVP 优先 PC 端，移动端后优化 |
| 执行阶段事件弹窗打断节奏 | 中 | 弹窗限时自动关闭，调优触发频率 |
| 数值不平衡导致太简单 / 太难 | 高 | 提供 debug 面板快速调参 |
| 存档数据结构频繁变更导致兼容问题 | 中 | 从 v0.1 就实现版本迁移机制 |
| Phaser 在低端移动设备性能不足 | 低 | 渲染降级策略（减少粒子、降低帧率） |

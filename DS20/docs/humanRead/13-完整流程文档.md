# PokeRogue 完整流程文档

本文档按源码实际执行顺序整理 `DS20/pokerogue-main` 的核心流程，只描述现有实现中的阶段、条件、队列顺序、分支和计算入口。

涉及的主要实现文件包括：

- `src/battle-scene.ts`
- `src/battle.ts`
- `src/phase-manager.ts`
- `src/dynamic-queue-manager.ts`
- `src/phases/*.ts`

## 1. 开局到首场战斗的启动流程

### 1.1 选完初始队伍后的建局

`SelectStarterPhase.start()` 在玩家确认初始队伍后执行以下流程：

1. 播放菜单 BGM，显示初始选择界面与存档槽界面。
2. 根据玩家选择，逐个创建初始队伍成员，写入以下数据：
   - 物种
   - 形态
   - 性别
   - 闪光
   - IV
   - 性格
   - 初始招式组
   - 被动能力开关
   - Pokerus
   - 昵称
   - 太晶属性
3. 应用初始挑战和初始队伍相关覆盖：
   - `overrideModifiers()`
   - `overrideHeldItems(party[0])`
4. 加载资源。
5. 淡出菜单。
6. 记录本局统计。
7. 调用 `globalScene.newBattle()` 创建当前战斗对象。
8. 调用 `globalScene.arena.init()` 初始化场地。
9. 重置局内计时数据。
10. 触发因已学招式导致的形态切换检查。
11. phase 结束。

### 1.2 `newBattle()` 的建局逻辑

`BattleScene.newBattle()` 的执行顺序如下：

1. 调用 `getNewBattleProps()` 计算新波次的基础参数。
2. `resetSeed(waveIndex)`，把本波的随机数种子重置到波次级别。
3. 决定战斗类型：
   - 读取存档恢复时走 `handleSavedBattle()`
   - 固定战走 `handleFixedBattle()`
   - 普通波次走 `handleNonFixedBattle()`
4. 计算本波是否是双打：
   - 先看 battle style override
   - 再看结局波、endless boss、ME 等硬性禁止双打条件
   - 再看固定战或训练家配置
   - 普通野战则按 `getDoubleBattleChance(waveIndex)` 随机
5. 保存上一战信息：
   - `lastEnemyTrainer`
   - `lastMysteryEncounter`
6. 在当前随机种子上下文中创建新的 `Battle` 实例。
7. 立即调用 `currentBattle.incrementTurn()`：
   - `turn` 加 1
   - 初始化 `turnCommands`
   - 初始化 `preTurnCommands`
   - 清空 `battleSeedState`
8. 如果这不是读档恢复，并且之前已经有上一场战斗，则执行 `doPostBattleCleanup(lastBattle, maxExpLevel)`。

### 1.3 `Battle` 构造时写入的核心字段

`Battle` 在构造时写入：

- `waveIndex`
- `battleType`
- `trainer`
- `mysteryEncounterType`
- `double`
- `battleSpec`
- `enemyLevels`

`battleSpec` 初始化规则：

- 经典模式且 `isWaveFinal(waveIndex)` 时，设为 `FINAL_BOSS`
- 其他情况为 `DEFAULT`

`enemyLevels` 初始化规则：

- 训练家战：`trainer.getPartyLevels(waveIndex)`
- 非训练家战：按当前战斗人数建立数组，并逐个调用 `getLevelForWave()`

### 1.4 野怪等级生成

`Battle.getLevelForWave()` 的计算过程：

1. 取难度波次 `levelWaveIndex = gameMode.getWaveForDifficulty(waveIndex)`。
2. 计算基础等级：

```text
baseLevel = 1 + levelWaveIndex / 2 + (levelWaveIndex / 25)^2
```

3. 如果当前波是 boss：

```text
ret = floor(baseLevel * 1.2)
```

4. 如果是最终 Boss 或 `waveIndex % 250 === 0`：

```text
level = ceil(ret / 25) * 25
```

5. 如果是普通 boss 且不是最终波：

```text
levelOffset = round(random(-1, 1) * floor(levelWaveIndex / 10))
level = ret + levelOffset
```

6. 如果不是 boss：
   - 计算 `deviation = 10 / levelWaveIndex`
   - 调用 `randSeedGaussForLevel(deviation)`
   - 结果取绝对值后加到 `baseLevel`

```text
level = max(round(baseLevel + abs(randSeedGaussForLevel(deviation))), 1)
```

`randSeedGaussForLevel(value)` 实现方式：

```text
rand = 0
循环 value 次:
  rand += randSeedFloat()
return rand / value
```

这里不是标准高斯分布实现，而是把多个 `randSeedFloat()` 相加后取平均。

## 2. 战后收尾到下一波预加载流程

### 2.1 `doPostBattleCleanup()` 的触发条件

当 `newBattle()` 不是从 session 读档恢复，且存在 `lastBattle` 时调用。

### 2.2 `doPostBattleCleanup()` 的执行顺序

1. 计算 `isNewBiome = this.isNewBiome(lastBattle)`。
2. 计算是否需要做 arena 级重置：

```text
resetArenaState =
  isNewBiome
  或 新战斗类型属于 TRAINER / MYSTERY_ENCOUNTER
  或 当前 battleSpec === FINAL_BOSS
```

3. 销毁上一战所有敌方对象。
4. 传播 Pokerus。
5. 如果不是新 biome 且下一战波次 `% 10 === 5`，更新昼夜池。
6. 取玩家场上应被收回的成员：
   - 按上一战是否双打，取前 1 或 2 个队伍成员
7. 若 `resetArenaState` 为真：
   - `arena.resetArenaEffects()`
   - 场上玩家单位移除 `COMMANDED`
   - 场上单位加入 `ReturnPhase`
   - 玩家全队 `resetBattleAndWaveData()`
   - 玩家全队 `resetTera()`
   - 对玩家全队应用 `PostBattleInitAbAttr`
   - 如果有 Terapagos，`arena.playerTerasUsed = 0`
   - 若主角立绘当前不可见，则加入 `ShowTrainerPhase`
8. 对玩家全队执行按昼夜触发的形态切换。
9. 根据 biome 是否变化决定下一步 encounter phase：
   - 不是随机 biome 且不是新 biome：`pushNew("NextEncounterPhase")`
   - 否则：`pushNew("NewBiomeEncounterPhase")`
10. 如果进入新 biome 且新的等级上限比旧值高，加入 `LevelCapPhase`

## 3. 遭遇创建与遭遇入场流程

### 3.1 遭遇 phase 的入口

首战由 `EncounterPhase` 进入。

后续波次由以下 phase 进入：

- 同 biome 连续波次：`NextEncounterPhase`
- 跨 biome 波次：`NewBiomeEncounterPhase`

三者的主要差别：

- `EncounterPhase`：开局首战或读档恢复战斗
- `NextEncounterPhase`：同 biome 连战，不重置 biome 天气和地形
- `NewBiomeEncounterPhase`：跨 biome 连战，按 arena transition 处理

### 3.2 `EncounterPhase.start()` 的顺序

1. `updateGameInfo()`
2. `initSession()`
3. 派发 `EncounterPhaseEvent`
4. 经典模式容错：
   - 如果 `waveIndex > 200`，直接 `GameOverPhase`
5. 若当前 battle 是 ME 且 `mysteryEncounter` 还未创建：
   - 基于波次偏移种子生成 encounter 实例
6. 若存在 ME：
   - 执行 `onInit()`
   - 填充对话 token
   - 加载 encounter 动画资源
   - 初始化 intro visuals
7. 遍历 `enemyLevels`，逐个生成或恢复敌人。

### 3.3 敌方单位生成规则

对于每个敌方槽位：

1. 如果当前是 ME，则跳过普通敌方生成，交给 ME 自己处理。
2. 如果不是读档恢复：
   - 训练家战：`trainer.genPartyMember(e)`
   - 野战：`randomSpecies(waveIndex, level, true)`
3. 若玩家持有 `BoostBugSpawnModifier`，且：
   - 当前波不是 boss
   - 当前 biome 不是 `END`
   - `randSeedInt(10) === 0`
   则把本次野怪替换为金虫网池中的物种。
4. 调用 `addEnemyPokemon(species, level, TrainerSlot.NONE, isBoss)` 创建敌人。
5. 经典最终 Boss 战时，敌方 IV 全设为 31。
6. 对玩家前排同步特性单位应用 `SyncEncounterNatureAbAttr`，把敌方性格同步到对应结果。
7. 若敌人是 Eternatus：
   - 经典最终 Boss 或最终波：设 boss、必要时切形态
   - endless 每 1000 波：切形态
8. 记录总种族值 `totalBst`，后面给双 boss 分段重分配使用。
9. 敌方资源预加载。
10. 若不是读档恢复，登记图鉴已见。

### 3.4 双 boss 分段重算

如果当前不是 ME，且：

- 没有 `ENEMY_HEALTH_SEGMENTS_OVERRIDE`
- 双打场上 boss 数量大于 1

则对每个 boss 重新计算血条段数：

```text
newSegments = ceil(originalSegments * (enemyBaseTotal / totalBst))
```

之后调用 `initBattleInfo()` 刷新战斗信息。

### 3.5 敌方 modifier 和 AI 初始化

在敌方资源加载完成后：

1. 如果不是读档恢复且不是 ME：
   - 根据战斗类型重建 modifier pool thresholds
   - 调用 `generateEnemyModifiers()`
   - `overrideModifiers(false)`
   - 对场上敌人执行 `overrideHeldItems(enemy, false)`
2. 如果是训练家战且 trainer 存在：
   - `trainer.genAI(globalScene.getEnemyParty())`

### 3.6 遭遇开始前的保存与天气地形设定

若不是读档恢复：

1. 先执行：
   - `trySetWeatherIfNewBiome()`
   - `trySetTerrainIfNewBiome()`
2. 再调用 `gameData.saveAll(...)`
3. 保存成功后：
   - `disableMenu = false`
   - 执行 `doEncounter()`
   - `resetSeed()`
4. 保存失败则 `reset(true)`

`saveAll(true, ...)` 的第二参数条件：

```text
battle.waveIndex % 5 === 1
或 (lastSavePlayTime >= 300)
```

## 4. 遭遇动画、文字和入场校正流程

### 4.1 普通 `doEncounter()`

`EncounterPhase.doEncounter()` 执行：

1. 播放当前战斗 BGM。
2. `updateModifiers(false)` 更新敌方 modifier 显示。
3. `setFieldScale(1)` 重置缩放。
4. 玩家全队 `resetWaveData()`。
5. 对以下对象做 2000ms 的平移入场：
   - `arenaEnemy`
   - 当前 trainer
   - `enemyField`
   - `arenaPlayer`
   - `trainer`
6. 如果没有 `battleSpec` 特殊流程，则进入 `doEncounterCommon()`。
7. 若有 ME intro visuals，同步移动其视觉层。

### 4.2 同 biome 的 `NextEncounterPhase.doEncounter()`

与普通 `doEncounter()` 不同的点：

1. 播放 BGM。
2. 玩家全队只执行 `resetWaveData()`，不做 arena 级重置。
3. 先把 `arenaNextEnemy` 设置为当前 biome。
4. 把上一波残留对象整体向右平移：
   - `arenaEnemy`
   - `arenaNextEnemy`
   - 当前 trainer
   - `enemyField`
   - `lastEnemyTrainer`
   - 上一场 ME 的 intro visuals
5. 平移完成后：
   - 把 `arenaEnemy` 对齐到 `arenaNextEnemy`
   - 隐藏 `arenaNextEnemy`
   - 销毁 `lastEnemyTrainer`
   - 移除上一场 ME visuals
6. 进入 `doEncounterCommon()`

`NextEncounterPhase` 明确覆盖了：

- `trySetWeatherIfNewBiome()`：空实现
- `trySetTerrainIfNewBiome()`：空实现

因此同 biome 连波不会重新设天气和场地地形。

### 4.3 新 biome 的 `NewBiomeEncounterPhase.doEncounter()`

执行顺序：

1. 播放 BGM。
2. 玩家全队 `resetBattleAndWaveData()`。
3. 若玩家单位仍在场，对其应用 `PostBiomeChangeAbAttr`。
4. 把：
   - `arenaEnemy`
   - `enemyField`
   - 当前 ME visuals
   向右平移 2000ms。
5. 完成后进入 `doEncounterCommon()`。

### 4.4 `doEncounterCommon()` 的分支

#### 4.4.1 野战

1. 敌方全部：
   - `untint`
   - `cry`
   - `showInfo`
2. 若敌方是闪光，验证成就。
3. 更新场地缩放。
4. 显示遭遇文本，文本结束后 phase 结束。

#### 4.4.2 训练家战

1. trainer 去 tint，播放动作。
2. 读取 `encounterMessages`。
3. 若有遭遇对白：
   - 可显示立绘
   - 播放对白
   - 对白结束后进入 `doSummon`
4. `doSummon` 中：
   - `currentBattle.started = true`
   - 播放 BGM
   - 显示双方球托盘
   - 隐藏敌方 trainer
   - `unshiftNew("SummonPhase", 0, false)`
   - 双打且敌方可出战人数大于 1 时再 `unshiftNew("SummonPhase", 1, false)`
5. phase 结束

#### 4.4.3 神秘事件

1. 播放 intro visuals 动画。
2. 若定义了 `onVisualsStart()`，则调用。
3. 否则若有 spriteConfigs，则播放闪光效果。
4. 显示开场文字与 intro dialogue。
5. 结束后：
   - 清理文字框标题
   - `unshiftNew("MysteryEncounterPhase")`
6. phase 结束

### 4.5 `EncounterPhase.end()` 结束时追加的队列

无论是野战还是训练家战，只要 `EncounterPhase.end()` 被执行，都会继续写入后续 phase。

执行顺序：

1. 对敌方场上单位：
   - 闪光则 `unshiftNew("ShinySparklePhase", battlerIndex)`
   - Eternatus 特殊处理时，把其可偷取持有物 modifier 改成不可转移
2. 若不是训练家战 / ME：
   - 若玩家有 `IvScannerModifier`
   - 则对每个敌方单位 `pushNew("ScanIvsPhase", battlerIndex)`
3. 若不是读档恢复：
   - 计算玩家可合法出战成员 `availablePartyMembers`
   - 如果 0 号位还没在场：`pushNew("SummonPhase", 0)`
   - 如果当前是双打：
     - 可合法人数大于 1：`pushNew("ToggleDoublePositionPhase", true)`
     - 1 号位未在场：`pushNew("SummonPhase", 1)`
   - 如果当前是单打：
     - 若 1 号位还在场：`pushNew("ReturnPhase", 1)`
     - `pushNew("ToggleDoublePositionPhase", false)`
   - 若不是训练家战，且不是 Daily 第 1 波：
     - 若合法人数大于当前最小上场人数：
       - `pushNew("CheckSwitchPhase", 0, currentBattle.double)`
       - 双打时再对 1 号位追加一次
4. `handleTutorial(Tutorial.ACCESS_MENU)` 完成后 `super.end()`
5. 无条件 `pushNew("InitEncounterPhase")`

## 5. 召唤、入场后处理和进入回合

### 5.1 `SummonPhase` 的流程

`SummonPhase.start()`：

1. 对即将上场的单位应用 `PreSummonAbAttr`
2. 调用 `preSummon()`

`preSummon()` 的分支如下。

#### 5.1.1 上场合法性修正

如果即将上场的队伍成员：

- 不允许出战
- 或者玩家侧该单位已不在队伍中

则：

1. 若该单位还在场，先 `leaveField()`
2. 从队伍中寻找位于当前索引之后的第一个合法成员
3. 若找不到：
   - `clearPhaseQueue()`
   - `unshiftNew("GameOverPhase")`
4. 若找到：
   - 交换当前槽位与合法槽位

#### 5.1.2 玩家召唤

1. 显示 `playerGo` 文本。
2. 隐藏球托盘。
3. 主角背面 sprite 做抛球动作。
4. 延迟 750ms 调用 `summon()`。

#### 5.1.3 训练家敌人召唤

1. 显示训练家派出文本。
2. 隐藏敌方球托盘。
3. 文本结束后调用 `summon()`。

#### 5.1.4 ME 野怪召唤

直接调用 `summonWild()`。

### 5.2 `summon()` 的具体顺序

1. 创建球体 sprite。
2. 设置上场位：
   - 右位固定 `FieldPosition.RIGHT`
   - 左位或中位按单双打及可用人数决定
3. 球体飞行动画。
4. 球体落点后：
   - 播放开球音效
   - 销毁球体
   - 把 pokemon 加入 field
   - 敌方则登记 `seenEnemyPartyMemberIds`
   - 更新 modifier
   - 更新场地缩放
   - `showInfo()`
   - `playAnim()`
   - 显示 sprite
   - 初始 scale 设为 0.5
   - tint 后再 untint
5. 缩放 tween 完成后：
   - `cry()`
   - 清 tint
   - `fieldSetup()`
   - 若 `summonData.speciesForm` 存在，则重新加载资源保持变身形态
6. 再延迟 1000ms 结束 phase。

### 5.3 `SummonPhase.onEnd()` 在 phase 结束时做的事

1. 若该单位是闪光：
   - `unshiftNew("ShinySparklePhase", battlerIndex)`
2. `pokemon.resetTurnData()`
3. 如果满足以下任一条件：
   - 不是读档恢复
   - 当前战斗类型属于 TRAINER / MYSTERY_ENCOUNTER
   - 当前波次 `% 10 === 1`
   则：
   - 触发 `SpeciesFormChangeActiveTrigger`
   - `queuePostSummon()`
4. `queuePostSummon()` 只做一件事：
   - `pokemon.turnData.summonedThisTurn = true`

### 5.4 `InitEncounterPhase`

`InitEncounterPhase.start()` 只做一轮扫描：

1. 遍历当前场上所有单位。
2. 若单位满足：
   - `pokemon.isEnemy()`
   - 或 `pokemon.turnData.summonedThisTurn`
   则：
   - `unshiftNew("PostSummonPhase", battlerIndex)`
3. phase 结束。

### 5.5 `PostSummonPhase`

`PostSummonPhase.start()` 的顺序：

1. 如果该单位中毒状态是 `TOXIC`，则把 `toxicTurnCount = 0`
2. `arena.applyTags(ArenaTagType.PENDING_HEAL, false, pokemon)`
3. `arena.applyTags(EntryHazardTag, false, pokemon)`
4. 若当前是 ME 且单位拥有 `MysteryEncounterPostSummonTag`：
   - `lapseTag(BattlerTagType.MYSTERY_ENCOUNTER_POST_SUMMON)`
5. 对所有友方单位应用 `CommanderAbAttr`
6. phase 结束

### 5.6 phase 队列为空时如何进入回合

`PhaseManager.shiftPhase()` 在队列为空时不会停止，而是直接调用 `turnStart()`。

`turnStart()` 的实现：

1. 清空动态 phase 队列
2. 把当前 phase 直接设成新的 `TurnInitPhase`
3. 立即开始执行

这意味着战斗过程中，只要 phase 队列自然跑空，就会自动进入新回合。

## 6. 回合初始化与指令录入流程

### 6.1 `TurnInitPhase.start()`

执行顺序：

1. 遍历玩家场上单位，检查是否因挑战限制而不再合法。
2. 若某单位不合法：
   - 入队提示文本
   - 计算全队合法单位
   - 若合法单位数为 0：
     - `clearPhaseQueue()`
     - `unshiftNew("GameOverPhase")`
   - 否则若后排足够替换：
     - `switchOut()`
   - 否则：
     - `leaveField()`
   - 若合法单位只剩 1 且当前是双打：
     - `unshiftNew("ToggleDoublePositionPhase", true)`
3. 派发 `TurnInitEvent`
4. 执行 `handleMysteryEncounterBattleStartEffects()`
5. 执行 `handleMysteryEncounterTurnStartEffects()`
   - 如果返回 `true`，当前 phase 直接结束，不继续排入 `CommandPhase`
6. 遍历当前场上所有 active 单位：
   - 玩家单位：
     - `currentBattle.addParticipant(playerPokemon)`
     - `resetTurnData()`
     - `pushNew("CommandPhase", battlerIndex)`
   - 敌方单位：
     - `resetTurnData()`
     - `pushNew("EnemyCommandPhase", battlerIndex - ENEMY_OFFSET)`
7. 最后 `pushNew("TurnStartPhase")`
8. phase 结束

### 6.2 `CommandPhase.start()`

对每个玩家上场位单独执行一次。

执行顺序：

1. `updateGameInfo()`
2. `resetCursorIfNeeded()`
3. 若 `fieldIndex > 0`，处理双打右位修正：
   - 如果场上其实只剩一个玩家单位，把 `fieldIndex` 改回中位
   - 如果左位已选择 `BALL` 或 `RUN`，本位 turnCommand 直接复制并 `skip = true`
4. 执行 `checkCommander()`
   - 若该单位正因 Commander 附着在友军身上，则本回合：
     - `command = FIGHT`
     - `move = NONE`
     - `skip = true`
5. 若本位 turnCommand 已经是 `skip`，直接结束 phase。
6. 尝试 `tryExecuteQueuedMove()`：
   - 清掉 move queue 前端不可用的招式
   - 若队列首项可直接执行，则直接写入 turnCommand，不再打开命令 UI
7. 如果当前是 ME 且 `skipToFightInput` 为真：
   - 直接进入 `UiMode.FIGHT`
8. 否则：
   - 进入 `UiMode.COMMAND`

### 6.3 `CheckSwitchPhase`

`CheckSwitchPhase` 只在非训练家战，且有额外可上场成员时进入。

终止条件依次为：

1. 当前 battle style 是 `SET`
2. 当前被检查单位已不在场
3. 队伍里没有其它 active 合法成员
4. 当前单位或队友存在禁止换人的状态：
   - `FRENZY`
   - `isTrapped()`
   - 任一玩家单位带 `COMMANDED`

如果上述条件都不满足：

1. 显示是否换人的确认文本。
2. 玩家确认后：
   - `unshiftNew("SwitchPhase", SwitchType.INITIAL_SWITCH, fieldIndex, false, true)`
3. 拒绝则直接结束。

## 7. 回合开始与行动排序流程

### 7.1 `TurnStartPhase.getCommandOrder()`

先取当前场上所有玩家和敌方 battler index。

排序规则：

1. 先按 command 类型排：
   - 非 `FIGHT` 指令优先于 `FIGHT`
   - 即 `SWITCH` / `BALL` / `RUN` 全都先于出招
2. 如果 command 类型相同，则维持原顺序。

这里 `getCommandOrder()` 只区分“非攻击动作优先于攻击动作”，不在这里完成最终招式先后比较。

### 7.2 `TurnStartPhase.start()` 的顺序

1. 计算 `moveOrder = getCommandOrder()`
2. 按 `inSpeedOrder(ArenaTagSide.BOTH)` 遍历全场：
   - 读取 `preTurnCommands`
   - 如果是 `Command.TERA`，则 `pushNew("TeraPhase", pokemon)`
3. 再按速度序遍历全场所有 `FIGHT` 指令单位：
   - 应用 `BypassSpeedChanceAbAttr`
   - 应用 `BypassSpeedChanceModifier`
4. 再按 `moveOrder` 遍历并调用 `handleTurnCommand()`
5. 所有 turnCommand 都排完后：
   - `pushNew("CheckInterludePhase")`
   - `queueTurnEndPhases()`
6. phase 结束

### 7.3 `handleTurnCommand()` 的分支

#### 7.3.1 `FIGHT`

调用 `handleFightCommand()`：

1. 从 `turnCommand.move` 取得排队招式。
2. 在当前 moveset 中寻找同 ID 且还有 PP 的招式实例。
3. 找不到则临时 new 一个 `PokemonMove(queuedMove.move)`。
4. 若招式有 `MoveHeaderAttr`：
   - `unshiftNew("MoveHeaderPhase", pokemon, move)`
5. `pushNew("MovePhase", pokemon, targets, move, useMode)`

#### 7.3.2 `BALL`

`unshiftNew("AttemptCapturePhase", targetFieldIndex, cursor)`

#### 7.3.3 `POKEMON`

`unshiftNew("SwitchSummonPhase", switchType, fieldIndex, cursor, true, pokemon.isPlayer())`

#### 7.3.4 `RUN`

`unshiftNew("AttemptRunPhase")`

### 7.4 `MovePhase` 在 phase 队列中的插队规则

`PhaseManager.unshiftPhase()` 对 `MovePhase` 做了特殊处理：

- 普通 `unshift` phase 直接插到当前队列前面
- `MovePhase` 不直接插到最前
- 它会被放到“下一个 `MoveEndPhase` 后面”

因此同一轮行动中，由招式引发的追击、连锁行动、额外行动不会打断当前 move 的清尾，而是在当前 move 的 `MoveEndPhase` 之后执行。

## 8. 单个行动的执行链

### 8.1 `MovePhase.start()` 的主要流程

`MovePhase` 的核心处理顺序为：

1. 检查使用者是否仍然 active。
2. 标记本回合已经行动。
3. 执行行动前失败检查。
4. 处理解冻等前置状态。
5. 处理忽略特性、重定向、反击目标等逻辑。
6. 扣 PP。
7. 处理行动前的形态切换。
8. 显示出招文本。
9. 再做一次失败检查。
10. 把本 battle 的 `lastMove` 设为当前招式。
11. 调用 `useMove(charging)`

`useMove()` 会把真正的效果 phase 入队：

```text
unshiftNew("MoveEffectPhase", userIndex, targets, move, useMode)
```

### 8.2 `MoveEndPhase`

`MoveEndPhase.start()` 的顺序：

1. 把 `pokemon.turnData.hitsLeft = -1`
2. 如果不是 follow-up move 且使用者仍 active：
   - `pokemon.lapseTags(AFTER_MOVE)`
3. `arena.setIgnoreAbilities(false)`
4. 对本次 targets 逐个应用 `PostSummonRemoveEffectAbAttr`
5. phase 结束

## 9. 倒地、补位和敌方换人的流程

### 9.1 `FaintPhase` 的进入点

当单位 HP 归零且未被其它分支拦截时进入 `FaintPhase`。

### 9.2 `FaintPhase` 的前置处理

进入后先处理：

- Destiny Bond 相关
- Grudge 相关
- 即时复活 modifier
- 参战者记录修正

如果是经典最终 Boss 第一形态的特殊倒地，不走普通 faint：

1. 敌人 HP 被强制回补到 `1`
2. 播放受击表现
3. 转入二阶段初始化逻辑

### 9.3 玩家单位倒地分支

玩家单位倒地后：

1. 若没有任何合法可战成员：
   - `GameOverPhase`
2. 否则若当前是双打，且场上只剩一个合法单位，并且该单位已经在场：
   - `ToggleDoublePositionPhase(true)`
3. 否则若后排存在合法单位：
   - `pushNew("SwitchPhase", SwitchType.SWITCH, fieldIndex, true, false)`

### 9.4 敌方单位倒地分支

敌方单位倒地后：

1. `unshiftNew("VictoryPhase", battlerIndex)`
2. 如果当前是训练家战或 ME 战斗，且敌方后排还有成员：
   - `pushNew("SwitchSummonPhase", SwitchType.SWITCH, fieldIndex, -1, false, false)`

### 9.5 倒地动画完成后的状态落地

faint 动画真正结束后：

1. 玩家单位失去友情：
   - `-FRIENDSHIP_LOSS_FROM_FAINT`
2. `doSetStatus(FAINT)`
3. 玩家单位从参战者集合中移除
4. 敌方单位增加分数并结算战利品：
   - `addBattleScore`
   - `addPostBattleLoot`
5. `leaveField()`

## 10. 敌方全灭后的结算链

### 10.1 `VictoryPhase` 的触发位置

每个敌方单位倒地时都会 `unshiftNew("VictoryPhase", battlerIndex)`。

### 10.2 `VictoryPhase.start()` 的核心顺序

1. 记录被击败数量。
2. 调用 `applyPartyExp(expValue, true)`。
3. 如果这不是 ME，且敌方 party 已经全部不能再战，则进入整场战斗结束分支。

### 10.3 整场战斗结束时 `VictoryPhase` 追加的 phase 顺序

敌方 party 全灭时：

1. `pushNew("BattleEndPhase", true)`
2. 如果是训练家战：
   - `pushNew("TrainerVictoryPhase")`
3. 如果不是最终通关：
   - `pushNew("EggLapsePhase")`
   - 如果经典固定 boss 波有固定奖励：
     - `pushNew("ModifierRewardPhase", ...)`
   - 如果 `currentWaveIndex % 10 !== 0`：
     - `pushNew("SelectModifierPhase", ..., custom settings?)`
   - 否则进入 X10 波奖励分支
   - 如果 `hasRandomBiomes` 或 `isNewBiome()`：
     - `pushNew("SelectBiomePhase")`
   - `pushNew("NewBattlePhase")`
4. 如果是最终通关：
   - `battleType = CLEAR`
   - 增加 clear score
   - `pushNew("GameOverPhase", true)`

### 10.4 X10 波奖励分支

#### 10.4.1 Daily

- 波次 10：给 `EXP_CHARM`
- 波次大于 10 且不是最终波：再给 `GOLDEN_POKEBALL`

#### 10.4.2 非 Daily

- endless 第 10 波：给 `EXP_SHARE`
- classic 第 10 波：给 `EXP_CHARM`
- 波次 `<= 750`，且满足：
  - `<= 500`
  - 或 `currentWaveIndex % 30 === superExpWave`
  则给经验 charm 类奖励
- 波次 `<= 150` 且是 50 的倍数：给 `GOLDEN_POKEBALL`
- endless 的 50 倍数波：
  - 给 voucher 奖励
  - `AddEnemyBuffModifierPhase`

## 11. `BattleEndPhase` 的战斗收尾

### 11.1 `BattleEndPhase.start()` 的顺序

1. 去重重复的 `BattleEndPhase`
2. 更新战斗统计
3. 若胜利：
   - endless 最高波更新
   - 结算 battle score
   - 训练家击败数更新
4. endless 若达到 `wave >= 5850`：
   - 清 phase 队列
   - `GameOverPhase(true)`
5. 对合法在场玩家单位应用 `PostBattleAbAttr`
6. 若有散落金钱：
   - `pickUpScatteredMoney()`
7. 清理敌方持有物 modifier
8. 销毁敌方 party 对象
9. 处理 lapsing modifier 的衰减与删除

## 12. 训练家胜利后的附加流程

### 12.1 `TrainerVictoryPhase.start()`

执行顺序：

1. 禁用菜单。
2. 播放胜利 BGM。
3. `unshiftNew("MoneyRewardPhase", trainerMoneyMultiplier)`
4. 对 trainer reward functions 逐个：
   - `unshiftNew("ModifierRewardPhase", ...)`
5. boss 训练家的 voucher 奖励也在这里插入。
6. 显示击败训练家文本。
7. 若 trainer 配置了胜利后对白，则继续显示对白。

### 12.2 `MoneyRewardPhase`

金额计算流程：

1. `moneyAmount = getWaveMoneyAmount(moneyMultiplier)`
2. 对 `moneyAmount` 应用 `MoneyMultiplierModifier`
3. 若 arena 存在 `HAPPY_HOUR`：

```text
moneyAmount *= 2
```

4. `addMoney(moneyAmount)`
5. 显示获得金钱文本

## 13. 蛋推进、商店选择和切 biome 的流程

### 13.1 `EggLapsePhase`

执行顺序：

1. 遍历所有蛋：
   - 如果 `EGG_IMMEDIATE_HATCH_OVERRIDE` 为真，立即可孵化
   - 否则先 `--egg.hatchWaves`
   - 小于 1 的进入本次孵化列表
2. 如果没有可孵化蛋，phase 结束。
3. 如果可孵化蛋数大于等于 2，且 `eggSkipPreference === 1`：
   - 提示是否跳过动画
   - 确认则静默孵化并进入 `EggSummaryPhase`
   - 否则逐个 `unshiftNew("EggHatchPhase", ...)`
4. 如果可孵化蛋数大于等于 2，且 `eggSkipPreference === 2`：
   - 直接静默孵化
   - `unshiftNew("EggSummaryPhase", eggHatchData)`
5. 其他情况：
   - 逐个 `unshiftNew("EggHatchPhase", ...)`

### 13.2 `SelectModifierPhase`

`SelectModifierPhase.start()` 的顺序：

1. 不是玩家控制时直接返回。
2. 首次进入且不是 copy：
   - `updateSeed()`
3. reroll 进入时：
   - `globalScene.reroll = false`
4. 若不是 copy：
   - `regenerateModifierPoolThresholds(party, poolType, rerollCount)`
5. 计算本次显示的奖励数量。
6. 生成 `typeOptions`。
7. 构建选择回调：
   - 跳过奖励
   - reroll
   - 转移持有物
   - 查看队伍
   - 锁 rarity
   - 选择奖励条目
   - 选择商店条目
8. `resetModifierSelect(...)` 打开 UI。

### 13.3 reroll 的处理

`rerollModifiers()` 中：

1. 计算 `rerollCost = getRerollCost(lockModifierTiers)`
2. 如果钱不够，报错。
3. `globalScene.reroll = true`
4. `unshiftNew("SelectModifierPhase", rerollCount + 1, 当前 rarity tiers)`
5. 退出当前 UI
6. 若没有费用豁免：
   - `money -= rerollCost`
7. 播放购买音效

### 13.4 `SelectBiomePhase`

执行顺序：

1. `resetSeed()`
2. 取：
   - `currentBiome`
   - `currentWaveIndex`
   - `nextWaveIndex = currentWaveIndex + 1`
3. 如果满足以下任一条件，下一 biome 直接为 `END`：
   - 经典且 `isWaveFinal(nextWaveIndex + 9)`
   - Daily 且 `isWaveFinal(nextWaveIndex)`
   - 短 biome 模式且 `nextWaveIndex % 50 === 0`
4. 如果是随机 biome 模式：
   - 直接 `generateNextBiome(nextWaveIndex)`
5. 如果当前 biome 有多条链接：
   - 过滤掉因权重未中过的分支
   - 如果剩余多于 1 条且玩家有 `MapModifier`
     - 打开 biome 选择 UI
   - 否则从剩余分支中随机选 1 个
6. 如果当前 biome 只有 1 条链接：
   - 直接使用该 biome
7. `setNextBiomeAndEnd(nextBiome)` 中：
   - 如果 `nextWaveIndex % 10 === 1`
     - 玩家应用 `MoneyInterestModifier`
     - 挑战允许全队恢复时：
       - `unshiftNew("PartyHealPhase", false)`
     - 否则：
       - `unshiftNew("SelectModifierPhase", ..., fixed battle custom settings?)`
   - `unshiftNew("SwitchBiomePhase", nextBiome)`
8. phase 结束

### 13.5 `SwitchBiomePhase`

切 biome 时：

1. 记录：
   - `lastEnemyTrainer`
   - `lastMysteryEncounter`
2. 淡出旧 arena
3. `globalScene.newArena(nextBiome)`
4. 交叉淡入新背景
5. 销毁旧 trainer

### 13.6 `PartyHealPhase`

执行顺序：

1. 淡出 BGM 和 UI。
2. 对全队做恢复：
   - 恢复 HP
   - 恢复状态
   - 恢复 PP
3. 如果挑战 `PREVENT_REVIVE` 生效：
   - 已倒地单位不会被复活
4. 重置信息显示。
5. `arena.playerTerasUsed = 0`

## 14. 神秘事件的完整流程

### 14.1 `MysteryEncounterPhase`

进入时执行：

1. `clearPhaseQueue()`，清掉标准战斗残留 phase
2. `encounter.updateSeedOffset()`
3. 若不是 followup option phase：
   - 把当前 encounter 记录进 `mysteryEncounterSaveData.encounteredEvents`
4. 设置 UI 到 `UiMode.MYSTERY_ENCOUNTER`

当玩家选择一个选项时：

1. 把选中的 option 写到 `selectedOption`
2. 若不是 followup：
   - 把 `selectedOption` 索引写入最近一条 ME save data
3. 若该 option 有 `onPreOptionPhase`：
   - 在 ME seed offset 下执行
4. 然后进入 `continueEncounter()`
5. `continueEncounter()` 会：
   - 显示 `option.dialogue.selected`
   - `pushNew("MysteryEncounterOptionSelectedPhase")`

### 14.2 `MysteryEncounterOptionSelectedPhase`

只做一件核心事情：

1. 在 ME seed offset 下执行所选 option 的 `onOptionPhase()`
2. 当前 phase 本身不决定后续链路
3. 后续 phase 必须由 `onOptionPhase()` 自己排入

### 14.3 `MysteryEncounterBattlePhase`

当某个选项把 ME 推入战斗时，执行该 phase。

处理顺序：

1. 根据 `encounterMode` 分为：
   - `WILD_BATTLE`
   - `BOSS_BATTLE`
   - `TRAINER_BATTLE`
2. 野战 / boss 战：
   - `unshiftNew("SummonPhase", 0, false)`
   - 双打且敌方可出战人数大于 1 时，再加 1 号位
   - 若不隐藏开场文本，显示 battle message
   - 然后进入 `endBattleSetup()`
3. 训练家战：
   - 显示 trainer
   - 播放对白
   - `currentBattle.started = true`
   - 播放 BGM
   - 显示双方球托盘
   - 敌方 trainer 退场
   - `unshiftNew("SummonPhase", 0, false)`
   - 双打再加 1 号位
   - 然后进入 `endBattleSetup()`

### 14.4 `MysteryEncounterBattlePhase.endBattleSetup()`

执行顺序：

1. 非 trainer battle 时，若玩家有 `IvScannerModifier`：
   - 对敌方依次 `pushNew("ScanIvsPhase")`
2. 检查玩家合法可战成员。
3. 若 0 号位不在场：
   - `pushNew("SummonPhase", 0)`
4. 双打时：
   - 合法成员数大于 1 则 `pushNew("ToggleDoublePositionPhase", true)`
   - 1 号位不在场时 `pushNew("SummonPhase", 1)`
5. 单打时：
   - 若 1 号位还在场：
     - 场上玩家单位依速度序移除 `COMMANDED`
     - `pushNew("ReturnPhase", 1)`
   - `pushNew("ToggleDoublePositionPhase", false)`
6. 非 trainer battle 且没有 `disableSwitch`：
   - 如果合法人数大于最小上场人数：
     - `pushNew("CheckSwitchPhase", 0, currentBattle.double)`
     - 双打时再对 1 号位追加
7. `pushNew("InitEncounterPhase")`
8. phase 结束

### 14.5 `MysteryEncounterBattleStartCleanupPhase`

这个 phase 用于处理 ME 战斗开始时，前一个流程残留的回合末状态。

执行顺序：

1. 仅衰减以下带 `TURN_END` lapse 的 tag：
   - `FLINCHED`
   - `ENDURING`
2. 移除所有 `PostTurnStatusEffectPhase`
3. 计算合法玩家单位与后排合法单位。
4. 若合法单位为 0：
   - `unshiftNew("GameOverPhase")`
5. 对每个场上不合法的玩家位：
   - 若后排有可替换成员，则 `unshiftNew("SwitchPhase", SwitchType.SWITCH, i, true, false)`
6. 若双打但只剩 1 只合法单位且后排为空：
   - `unshiftNew("ToggleDoublePositionPhase", true)`
7. 场上所有单位 `resetTurnData()`

### 14.6 `MysteryEncounterRewardsPhase`

执行顺序：

1. 若 encounter 定义了 `doContinueEncounter()`：
   - 直接执行它
2. 否则在 `waveIndex * 1000` 的 seed offset 下：
   - 执行 `encounter.onRewards()`
   - 然后进入 `doEncounterRewardsAndContinue()`
3. `doEncounterRewardsAndContinue()` 中：
   - 若有 `doEncounterExp()`，则执行
   - 若有 `doEncounterRewards()`，则执行
   - 否则若 `addHealPhase` 为真：
     - 删掉所有 `SelectModifierPhase`
     - `unshiftNew("SelectModifierPhase", 0, undefined, { fillRemaining: false, rerollMultiplier: -1 })`
   - `pushNew("PostMysteryEncounterPhase")`

### 14.7 `PostMysteryEncounterPhase`

执行顺序：

1. 若当前 option 有 `onPostOptionPhase`：
   - 在 ME seed offset 下执行
2. 然后进入 `continueEncounter()`
3. `continueEncounter()` 中：
   - 显示 outro dialogue
   - 若 `hasRandomBiomes` 或 `isNewBiome()`：
     - `pushNew("SelectBiomePhase")`
   - `pushNew("NewBattlePhase")`

## 15. 回合末流程

### 15.1 `queueTurnEndPhases()` 固定顺序

`PhaseManager.queueTurnEndPhases()` 固定按以下顺序压入：

1. `WeatherEffectPhase`
2. `PositionalTagPhase`
3. `BerryPhase`
4. `CheckStatusEffectPhase`
5. `TurnEndPhase`

### 15.2 `CheckInterludePhase`

在 `TurnStartPhase` 末尾先于回合末队列被压入。

如果满足 interlude 条件：

1. 调用 `phaseManager.onInterlude()`
2. `onInterlude()` 会：
   - 移除 `WeatherEffectPhase`
   - 移除 `BerryPhase`
   - 移除 `CheckStatusEffectPhase`
   - 把现有 `TurnEndPhase.upcomingInterlude = true`

### 15.3 `WeatherEffectPhase`

对于伤害类天气：

1. 检查天气是否被抑制
2. 对每个 battler：
   - 若不免疫
   - 不在 underground / underwater
   - 未被阻挡
   则造成：

```text
weatherDamage = maxHp / 16
```

3. 显示天气流逝文本
4. 应用 `PostWeatherLapseAbAttr`
5. 播放公共动画

### 15.4 `BerryPhase`

每个 battler：

1. 一次性消费所有当前可触发的树果
2. 敌方可能通过 `PreventBerryUseAbAttr` 阻止食用
3. 食用后触发 `CudChewConsumeBerryAbAttr`

### 15.5 `CheckStatusEffectPhase`

按速度序检查每个 battler：

1. 若状态是回合末伤害型状态：
   - `POISON`
   - `TOXIC`
   - `BURN`
2. 则 `unshiftNew("PostTurnStatusEffectPhase", battlerIndex)`

### 15.6 `PostTurnStatusEffectPhase`

执行顺序：

1. 检查单位仍 active，且不是切出状态。
2. `status.incrementTurn()`
3. 先检查：
   - `BlockNonDirectDamageAbAttr`
   - `BlockStatusDamageAbAttr`
4. 计算伤害：

普通中毒：

```text
damage = maxHp / 8
```

剧毒：

```text
damage = maxHp * toxicTurnCount / 16
```

灼伤：

```text
damage = maxHp / 16
```

灼伤还会额外应用 `ReduceBurnDamageAbAttr`。

5. 造成伤害并播放动画。
6. 若因此触发最终 Boss 二阶段，则调用相应初始化逻辑。

### 15.7 `TurnEndPhase`

执行顺序：

1. `currentBattle.incrementTurn()`
2. 派发 `TurnEndEvent`
3. 清空上一轮行动顺序记录
4. 隐藏 ability bar
5. 遍历全场 battler，跳过正在切出的单位：
   - `lapseTags(TURN_END)`
   - 应用 `TurnHealModifier`
   - 草场回血逻辑
   - 敌方回合末回血 / 解异常逻辑
   - `PostTurnAbAttr`
   - `TurnStatusEffectModifier`
   - `TurnHeldItemTransferModifier`
   - 累加临时 turn counters
6. 如果不是 `upcomingInterlude`：
   - `arena.lapseTags()`
7. 衰减天气和场地持续回合
8. 持续时间归零时清除天气和场地

## 16. 失败、重试和通关流程

### 16.1 `GameOverPhase.start()` 的入口分支

进入后顺序：

1. 隐藏 ability bar
2. 经典模式若 `waveIndex > 200`，强制视为胜利
3. 若当前是 ME 且 `mysteryEncounter.onGameOver()` 返回 `false`：
   - 不结束游戏
   - 当前 phase 直接结束
4. 若是 endless 胜利：
   - 播放 ending dialogue
   - 然后进入 `handleGameOver()`
5. 若是胜利，或未开启 retries：
   - 直接 `handleGameOver()`
6. 否则进入重试确认

### 16.2 失败后的 retry 流程

玩家确认 retry 时：

1. UI 淡出
2. `globalScene.reset()`
3. `phaseManager.clearPhaseQueue()`
4. `gameData.loadSession(sessionSlotId)`
5. 重新入队：
   - `pushNew("EncounterPhase", true)`
   - `pushNew("SummonPhase", 0, true, true)`
   - 如果双打且可战人数大于 1，再加 1 号位
   - 若 `waveIndex > 1` 且不是训练家战：
     - `pushNew("CheckSwitchPhase", 0, currentBattle.double)`
     - 双打且可战人数大于 1 时再加 1 号位
6. UI 淡入
7. 当前 `GameOverPhase` 结束

### 16.3 真正结算的 `handleGameOver()`

执行顺序：

1. 调服务端或本地接口确认是否是 new clear。
2. 若胜利：
   - 经典模式发放经典通关成就、统计和 ribbon
   - Daily 首次通关发放 daily 胜利统计和成就
3. 计算淡出时长：
   - 胜利 10000ms
   - 失败 5000ms
4. 淡出 BGM。
5. 隐藏场上 active 单位信息。
6. UI 淡出。
7. 清空 phase 队列。
8. 清空文本。
9. 若胜利且是 challenge 模式：
   - 结算挑战成就
10. 若胜利且是经典模式：
    - 可能播放 ending dialogue
    - 创建 `EndCardPhase`
11. 记录 run history。
12. `pushNew("PostGameOverPhase", sessionSlotId, endCardPhase?)`

### 16.4 胜利时的 unlock 和奖励 phase

经典通关时，`handleUnlocks()` 可能 `unshiftNew("UnlockPhase", ...)` 解锁：

- `ENDLESS_MODE`
- `SPLICED_ENDLESS_MODE`
- `MINI_BLACK_HOLE`
- `EVIOLITE`

若存在首次 ribbon 完成，还会插入：

- `RibbonModifierRewardPhase`
- `GameOverModifierRewardPhase`

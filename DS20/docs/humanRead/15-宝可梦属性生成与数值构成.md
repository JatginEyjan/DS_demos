# PokeRogue 宝可梦属性生成与数值构成

本文档描述 `DS20/pokerogue-main` 中一只宝可梦在创建时，静态属性来自哪里，随机属性如何生成，最终数值如何计算。

## 1. 静态物种属性来自哪里

物种静态数据的主来源：

- `src/data/balance/pokemon-species.ts`

运行时类型定义：

- `src/data/pokemon-species.ts`

初始化路径：

```text
initializeGame()
-> initSpecies()
-> allSpecies.push(new PokemonSpecies(...))
```

也就是说，所有宝可梦基础模板都由 `initSpecies()` 把 `PokemonSpecies` / `PokemonForm` 实例压入 `allSpecies`。

## 2. `PokemonSpecies` 保存的核心静态字段

`PokemonSpeciesForm` / `PokemonSpecies` 体系中，核心静态字段包括：

- `speciesId`
- `generation`
- `type1`
- `type2`
- `height`
- `weight`
- `ability1`
- `ability2`
- `abilityHidden`
- `baseTotal`
- `baseStats`
- `catchRate`
- `baseFriendship`
- `baseExp`
- `growthRate`
- `malePercent`
- `genderDiffs`
- `isStarterSelectable`
- `forms`

`baseStats` 以数组保存，顺序固定为：

```text
[HP, ATK, DEF, SPATK, SPDEF, SPD]
```

## 3. 一只宝可梦创建时的主流程

`Pokemon` 构造函数的关键流程如下：

1. 写入 `species`
2. 写入 `level`
3. `abilityIndex = 传入值 ?? generateAbilityIndex()`
4. 若外部传入 `formIndex / gender / shiny / variant`，则先直接写入
5. `exp = dataSource?.exp || getLevelTotalExp(level, species.growthRate)`
6. 如果存在 `dataSource`：
   - 直接从存档或数据源恢复现有属性
7. 否则进入“新生成宝可梦”流程：
   - `id = randSeedInt(4294967296)`
   - `ivs = 传入ivs || getIvsFromId(id)`
   - 若没传 gender：`species.generateGender()`
   - 若没传 form：`getSpeciesFormIndex(species, gender, nature, isPlayer())`
   - 若没传 shiny：`trySetShiny()`
   - 若没传 variant：闪光时 `generateShinyVariant()`，否则 `0`
   - 若传了 nature：`setNature(nature)`
   - 否则：`generateNature()`
   - `friendship = species.baseFriendship`
   - `metLevel = level`
   - `metBiome = 当前 biome 或 -1`
   - `metSpecies = species.speciesId`
   - `metWave = 当前 wave 或 -1`
   - `pokerus = false`
   - level > 1 时可能触发 fusion 生成
   - `luck = shiny/fusion shiny 对应 luck`
   - `teraType = randSeedItem(this.getTypes(false, false, true))`
8. 最后：
   - `generateName()`
   - 对新生成对象调用 `calculateStats()`

## 4. 物种模板字段是怎么写进代码的

`initSpecies()` 中每个 `PokemonSpecies(...)` 的参数顺序就是该物种基础模板的来源。

以普通 `PokemonSpecies` 为例，构造参数中直接写入：

1. 物种 ID
2. 世代
3. 是否二级神 / 一级神 / 幻兽
4. 图鉴类别名
5. 主属性 / 副属性
6. 身高 / 体重
7. 第一特性 / 第二特性 / 隐藏特性
8. 种族值总和
9. 六围基础值
10. 捕获率
11. 基础亲密度
12. 基础经验
13. 成长率
14. 雄性概率
15. 性别差异开关
16. 是否可作为 starter
17. form 列表

`PokemonForm(...)` 里也会重复写 form 自己的类型、特性、种族值、外形尺寸等。

## 5. 性别是怎么生成的

`PokemonSpecies.generateGender()` 的规则：

1. `malePercent == null`
   - 返回 `GENDERLESS`
2. 否则：

```text
if randSeedFloat() * 100 <= malePercent:
  MALE
else:
  FEMALE
```

也就是说，性别只依赖物种模板上的 `malePercent`。

## 6. 特性索引是怎么生成的

`Pokemon.generateAbilityIndex()` 的规则：

1. 初始隐藏特性概率：

```text
hiddenAbilityChance = BASE_HIDDEN_ABILITY_CHANCE
```

2. `BASE_HIDDEN_ABILITY_CHANCE` 定义于 `src/data/balance/rates.ts`

```text
256 / 65536 = 1 / 256
```

3. 如果不是 trainer 宝可梦：
   - 对 `hiddenAbilityChance` 应用 `HiddenAbilityRateBoosterModifier`
4. 普通特性位：

```text
regularAbility =
  species.ability2 !== species.ability1 ? randSeedInt(2) : 0
```

5. 是否使用隐藏特性：

```text
useHiddenAbility =
  species.abilityHidden ? !randSeedInt(hiddenAbilityChance.value) : false
```

6. 返回：

```text
useHiddenAbility ? 2 : regularAbility
```

因此：

- `0` = 第一特性
- `1` = 第二特性
- `2` = 隐藏特性

## 7. IV 是怎么生成的

### 7.1 普通新生成宝可梦

默认 IV 来源：

```text
ivs = getIvsFromId(id)
```

`getIvsFromId(id)` 定义在 `src/utils/common.ts`，做法是把 32 位 `id` 拆成 6 段 5-bit：

```text
HP    = (id & 0x3e000000) >>> 25
ATK   = (id & 0x01f00000) >>> 20
DEF   = (id & 0x000f8000) >>> 15
SPATK = (id & 0x00007c00) >>> 10
SPDEF = (id & 0x000003e0) >>> 5
SPD   =  id & 0x0000001f
```

因此单项 IV 范围固定为 `0-31`。

### 7.2 训练家敌人的 IV

`EnemyPokemon` 在 trainer 战中会覆盖默认 IV。

规则：

```text
while ivs.length < 6:
  ivs.push(randSeedIntRange(floor(waveIndex / 10), 31))
```

也就是 trainer 敌人的单项 IV 最低值会随波次抬高：

```text
minimumIv = floor(waveIndex / 10)
maximumIv = 31
```

### 7.3 最终 Boss 特殊 IV

经典最终 Boss 战，`EncounterPhase.start()` 中会把敌方 IV 直接全设成 `31`。

## 8. 性格是怎么生成的

### 8.1 默认流程

如果构造时没有传入 `nature`：

```text
generateNature()
```

实现：

1. 若没给 `naturePool`，则使用全部 25 个 Nature
2. `nature = naturePool[randSeedInt(naturePool.length)]`
3. `setNature(nature)`

### 8.2 `setNature()`

`setNature(nature)` 会：

1. 写入 `this.nature`
2. 立即 `calculateStats()`

所以性格变更会立即重算六围。

## 9. 闪光是怎么生成的

### 9.1 普通闪光判定

`trySetShiny()` 的规则：

1. endless + `END` biome 时不允许闪光
2. 取 Pokémon ID 的高 16 位和低 16 位：

```text
rand1 = (id & 0xffff0000) >>> 16
rand2 =  id & 0x0000ffff
```

3. 取玩家训练家 ID 和 secret ID：

```text
E = trainerId ^ secretId
F = rand1 ^ rand2
```

4. 初始阈值：

```text
shinyThreshold = BASE_SHINY_CHANCE
```

`BASE_SHINY_CHANCE = 64`，对应：

```text
64 / 65536 = 1 / 1024
```

5. 若没有手动覆盖阈值：
   - 活动期间可能提高 trainer / wild 的闪率
   - 玩家方或野怪会应用 `ShinyRateBoosterModifier`
6. 判定：

```text
shiny = (E ^ F) < shinyThreshold
```

### 9.2 基于种子的二次闪光判定

`trySetShinySeed()` 会改用：

```text
randSeedInt(65536) < shinyThreshold
```

适用于需要再次 roll 闪光的场景。

## 10. 闪光 variant 是怎么生成的

闪光 variant 相关常量定义在 `src/data/balance/rates.ts`：

- `SHINY_VARIANT_CHANCE = 4`
- `SHINY_EPIC_CHANCE = 1`

注释说明：

- 先判定 shiny 是否成为 variant，概率是 `x/10`
- 若成为 variant，再判定是否是 epic variant

运行时 `variant` 的含义：

- `0` = 默认闪光变体
- `1` = 第二变体
- `2` = 第三变体

图鉴位图对应：

- `DEFAULT_VARIANT`
- `VARIANT_2`
- `VARIANT_3`

## 11. 亲密度是怎么初始化的

### 11.1 普通宝可梦

普通创建时：

```text
friendship = species.baseFriendship
```

### 11.2 训练家敌人

训练家敌人在 `EnemyPokemon` 中覆盖：

```text
friendship = clamp(
  round(255 * (waveIndex / TRAINER_MAX_FRIENDSHIP_WAVE)),
  TRAINER_MIN_FRIENDSHIP,
  255
)
```

也就是 trainer 敌人的亲密度随波次线性抬升，并受最小/最大边界限制。

## 12. 太晶属性是怎么生成的

普通新生成宝可梦：

```text
teraType = randSeedItem(this.getTypes(false, false, true))
```

也就是从当前可用类型集合里随机选一个。

## 13. 经验值与等级是怎么初始化的

创建时：

```text
exp = getLevelTotalExp(level, species.growthRate)
levelExp = 0
```

也就是说，新生成的同等级宝可梦，初始总经验总是对齐到该等级经验下限。

## 14. 成长率与等级经验来自哪里

经验表来源：

- `src/data/exp.ts`

成长率枚举：

- `ERRATIC`
- `FAST`
- `MEDIUM_FAST`
- `MEDIUM_SLOW`
- `SLOW`
- `FLUCTUATING`

`getLevelTotalExp(level, growthRate)` 的规则：

### 14.1 1-99 级

优先从 `expLevels[growthRate][level - 1]` 读表。

如果成长率不是 `MEDIUM_FAST`，还会额外做一次混合：

```text
floor(levelExp * 0.325 + mediumFastExp(level) * 0.675)
```

### 14.2 100 级及以上

按成长率公式现算，然后若不是 `MEDIUM_FAST`，仍做：

```text
floor(ret * 0.325 + mediumFastExp(level) * 0.675)
```

## 15. 最终六围是怎么计算的

`calculateStats()` 的流程：

1. 先取 `baseStats = calculateBaseStats()`
2. 然后对 6 项永久属性逐项计算

### 15.1 HP

基础公式：

```text
floor((2 * baseHp + ivHp) * level * 0.01) + level + 10
```

随后：

1. 应用 `PokemonIncrementingStatModifier`
2. 若拥有 `WONDER_GUARD` 特殊分支，则 HP 直接设为 `1`
3. 若新最大 HP 变高，当前 HP 也会同步补差值

### 15.2 非 HP 五围

基础公式：

```text
floor((2 * baseStat + iv) * level * 0.01) + 5
```

然后：

1. 读取 `getNatureStatMultiplier(nature, stat)`
2. 应用 `PokemonNatureWeightModifier`
3. 若倍率不为 1：
   - 提升时 `ceil(stat * multiplier)`
   - 降低时 `floor(stat * multiplier)`
   - 最低不低于 `1`
4. 再应用 `PokemonIncrementingStatModifier`

### 15.3 最终裁剪

每项最终都做：

```text
Clamp(value, 1, Number.MAX_SAFE_INTEGER)
```

## 16. `calculateBaseStats()` 的来源与修正顺序

`calculateBaseStats()` 的顺序：

1. 从当前 form 取基础六围：

```text
baseStats = getSpeciesForm(true).baseStats.slice(0)
```

2. 应用挑战：
   - `ChallengeType.FLIP_STAT`
3. 应用 modifier：
   - `PokemonBaseStatTotalModifier`
   - `PokemonBaseStatFlatModifier`
4. 如果是 fusion：

```text
baseStats[s] = ceil((baseStats[s] + fusionBaseStats[s]) / 2)
```

5. 如果不是 fusion，但当前模式是 `SplicedOnly`：

```text
baseStats[s] = ceil(baseStats[s] / 2)
```

6. 最后再应用 `BaseStatModifier`

因此最终数值顺序不是“物种模板直接套公式”，而是：

```text
物种基础值
-> challenge
-> 总量/平面修正
-> fusion / spliced only
-> vitamins / base stat modifier
-> 等级+IV公式
-> 性格倍率
-> 增量 modifier
```

## 17. 融合时属性如何处理

`calculateBaseStats()` 里对融合的处理是逐项平均后向上取整：

```text
fusionBaseStat = ceil((baseStatA + baseStatB) / 2)
```

这一步发生在进入最终六围公式之前。

## 18. 玩家与敌人在生成上的差异

### 18.1 玩家侧

- 通常使用 `getIvsFromId(id)`
- 性格纯随机或使用外部指定
- 可受 starter 选择、存档、蛋、固定配置影响

### 18.2 野怪

- 同样默认 `getIvsFromId(id)`
- 特性可受隐藏特性概率和 modifier 影响
- 闪光可受活动和 shiny booster 影响

### 18.3 训练家敌人

- IV 改为按波次下限随机
- 亲密度按波次增长
- 招式使用 trainer moveset generation
- 某些 Daily / Boss 配置可强行覆盖 nature、moveset、form、variant

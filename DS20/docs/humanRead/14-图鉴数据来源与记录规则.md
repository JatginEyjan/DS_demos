# PokeRogue 图鉴数据来源与记录规则

本文档只描述 `DS20/pokerogue-main` 中图鉴数据的来源、存储结构、初始化规则、更新规则和解锁位图。

## 1. 图鉴数据的来源

图鉴系统的数据来源分成两层：

1. 物种静态数据
   - 来源文件：`src/data/balance/pokemon-species.ts`
   - 载入容器：`src/data/data-lists.ts` 中的 `allSpecies`
   - 初始化入口：`src/init/init.ts -> initializeGame() -> initSpecies()`
2. 玩家图鉴进度数据
   - 来源类型：`src/@types/dex-data.ts`
   - 持久化实现：`src/system/game-data.ts`
   - 运行时字段：`GameData.dexData`

静态物种数据决定“图鉴里可以显示什么”；`dexData` 决定“玩家已经看见/捕获/解锁了什么”。

## 2. 图鉴的运行时结构

`DexData` 的定义：

```ts
export interface DexData {
  [key: number]: DexEntry;
}
```

`DexEntry` 的定义：

```ts
export interface DexEntry {
  seenAttr: bigint;
  caughtAttr: bigint;
  natureAttr: number;
  seenCount: number;
  caughtCount: number;
  hatchedCount: number;
  ivs: number[];
  ribbons: RibbonData;
}
```

也就是说，图鉴以 `speciesId` 为 key，每个物种有一条 `DexEntry`。

## 3. `dexData` 的初始化规则

`GameData` 构造时会调用：

1. `initDexData()`
2. `initStarterData()`

### 3.1 `initDexData()` 的基础初始化

`initDexData()` 会遍历 `allSpecies`，为每个 `speciesId` 建一条初始记录：

- `seenAttr = 0n`
- `caughtAttr = 0n`
- `natureAttr = 0`
- `seenCount = 0`
- `caughtCount = 0`
- `hatchedCount = 0`
- `ivs = [0, 0, 0, 0, 0, 0]`
- `ribbons = new RibbonData(0)`

### 3.2 默认初始御三家图鉴

`initDexData()` 还会对 `defaultStarterSpecies` 做一轮默认解锁。

默认 starter 图鉴属性：

```text
NON_SHINY | MALE | FEMALE | DEFAULT_VARIANT | DEFAULT_FORM
```

默认 starter 性格：

1. 用固定 seed：`executeWithSeedOffset(..., 0, "default")`
2. 从中性性格池中随机：
   - `HARDY`
   - `DOCILE`
   - `SERIOUS`
   - `BASHFUL`
   - `QUIRKY`
3. 每个默认 starter 只会写入一个默认性格位

默认 starter IV：

- 六项全部写成 `15`

最终 `defaultDexData` 会保存这一版默认值，供后续缺省迁移使用。

## 4. 图鉴位图的来源

图鉴使用 `DexAttr` 作为位图基础定义，位置如下：

- `src/enums/dex-attr.ts`

定义如下：

```ts
NON_SHINY: 1n
SHINY: 2n
MALE: 4n
FEMALE: 8n
DEFAULT_VARIANT: 16n
VARIANT_2: 32n
VARIANT_3: 64n
DEFAULT_FORM: 128n
```

form 位不是写死常量，而是通过 `getFormAttr(formIndex)` 动态生成：

```text
getFormAttr(formIndex) = 1n << (7 + formIndex)
```

因此：

- `formIndex = 0` -> `128`
- `formIndex = 1` -> `256`
- `formIndex = 2` -> `512`
- `formIndex = 3` -> `1024`

## 5. 单只宝可梦如何转换成图鉴位图

`Pokemon.getDexAttr()` 负责把一只运行时宝可梦转换成图鉴属性位图。

规则如下：

1. 若不是无性别：
   - 雄性写 `MALE`
   - 雌性写 `FEMALE`
2. 闪光：
   - 闪光写 `SHINY`
   - 非闪光写 `NON_SHINY`
3. 变体：
   - `variant = 0` -> `DEFAULT_VARIANT`
   - `variant = 1` -> `VARIANT_2`
   - `variant >= 2` -> `VARIANT_3`
4. 形态：
   - 通过 `getFormAttr(formIndex)` 写入

## 6. 物种可解锁上限是如何计算的

`PokemonSpecies.getFullUnlocksData()` 会生成该物种“理论上可解锁的最大位图”。

计算顺序：

1. 一定加入：
   - `NON_SHINY`
   - `SHINY`
2. 若有性别：
   - `malePercent > 0` 时加入 `MALE`
   - `malePercent < 100` 时加入 `FEMALE`
3. 一定加入：
   - `DEFAULT_VARIANT`
4. 若实现了额外变体：
   - 加入 `VARIANT_2`
   - 加入 `VARIANT_3`
5. 形态位：
   - 若 `forms.length > 1`
   - 对每个非 `isUnobtainable` form，加入对应 form bit
   - 否则加入 `DEFAULT_FORM`

`setPokemonCaught()` 在写图鉴时会做一次：

```text
dexAttr = pokemon.getDexAttr() & species.getFullUnlocksData()
```

所以不能解锁的位会在写入前被遮掉。

## 7. `setPokemonSeen()` 的更新规则

`GameData.setPokemonSeen(pokemon, incrementCount = true, trainer = false)` 的行为：

1. 如果当前是 ME 且 encounter 设置了 `preventGameStatsUpdates`：
   - 直接返回，不更新
2. 取 `dexEntry = dexData[pokemon.species.speciesId]`
3. 执行：

```text
dexEntry.seenAttr |= pokemon.getDexAttr()
```

4. 如果 `incrementCount = true`：
   - `dexEntry.seenCount++`
   - `gameStats.pokemonSeen++`
   - 非 trainer 时再按神兽类别累加 seen 统计
   - 若是闪光，再累加 `shinyPokemonSeen`

注意：

- `seenAttr` 用当前这只 Pokémon 的物种 ID 直接写
- 不做前置形态追溯
- 不做 pre-evo 追溯

## 8. `setPokemonCaught()` 的更新规则

`GameData.setPokemonCaught()` 分两层：

1. `setPokemonCaught(pokemon, ...)`
2. `setPokemonSpeciesCaught(pokemon, species, ...)`

### 8.1 非真实捕获时的保护条件

如果 `incrementCount = false`，且该物种根形态从未被捕获：

```text
if (!dexData[speciesRootForm].caughtAttr) return false
```

这条规则会阻止“租借宝可梦”“临时状态变化”去污染玩家还没真正抓过的物种图鉴。

### 8.2 捕获写入主流程

`setPokemonSpeciesCaught()` 的顺序：

1. 取 `dexEntry = dexData[species.speciesId]`
2. 保存旧值 `caughtAttr`
3. 取 `formIndex = pokemon.formIndex`
4. 计算当前 Pokémon 对应的图鉴位图：

```text
dexAttr = pokemon.getDexAttr() & species.getFullUnlocksData()
```

5. 写入：

```text
dexEntry.caughtAttr |= dexAttr
```

### 8.3 战斗形态补写基础形态

如果抓到的是非 0 号 form，还会补写基础 form。

特殊分支：

- Pikachu 以 Pichu 根形态解锁时，补写 form 0
- Urshifu：
  - `formIndex = 2` -> 补 form 0
  - `formIndex = 3` -> 补 form 1
- Zygarde：
  - `formIndex = 4` -> 补 form 2
  - `formIndex = 5` -> 补 form 3

通用分支：

- 若当前 formKey 在 `pokemonFormChanges` 中存在进入当前 form 的 form change 规则
- 则补写 `form 0`

### 8.4 starter ability 解锁

如果该物种存在于 `speciesStarterCosts`：

```text
starterData[speciesId].abilityAttr |=
  pokemon.abilityIndex !== 1 || pokemon.species.ability2
    ? 1 << pokemon.abilityIndex
    : AbilityAttr.ABILITY_HIDDEN
```

也就是：

- 一般情况下直接按 `abilityIndex` 写入能力位
- 但当 `abilityIndex === 1` 且物种没有第二特性时，把它视为隐藏特性位

### 8.5 性格解锁

每次真正写入 caught 时都会执行：

```text
dexEntry.natureAttr |= 1 << (pokemon.nature + 1)
```

也就是 `natureAttr` 保存的是“已经抓到过哪些性格”。

### 8.6 计数更新

如果 `incrementCount = true`：

- `fromEgg = true`
  - `hatchedCount++`
  - 更新 hatched 相关 gameStats
- `fromEgg = false`
  - `caughtCount++`
  - 更新 caught 相关 gameStats

### 8.7 新捕获与新属性判断

源码里还会判断：

```text
newCatch = !caughtAttr
hasNewAttr = (caughtAttr & dexAttr) !== dexAttr
```

含义分别是：

- `newCatch`：之前完全没抓到过这个物种
- `hasNewAttr`：虽然抓过，但这次抓到了新的 shiny / 性别 / 变体 / form 组合

## 9. `natureAttr` 的使用方式

`natureAttr` 是一个普通 number 位图。

写入规则：

```text
1 << (nature + 1)
```

读取规则：

- `getSpeciesDefaultNature()`：返回第一个被置位的性格
- `getNaturesForAttr()`：遍历 25 个性格位，把所有已解锁性格返回

补充规则：

`consolidateDexData()` 在迁移旧存档时，如果：

- `natureAttr` 不存在
- 或 `caughtAttr` 已有值但 `natureAttr` 为空

则：

- 优先用 `defaultDexData` 中的默认值
- 否则随机补一个性格位

## 10. 图鉴 IV 记录规则

`updateSpeciesDexIvs(speciesId, ivs)` 的行为：

1. 取当前 `dexEntry.ivs`
2. 对六项逐项比较
3. 如果本次 `ivs[i]` 更高，则覆盖保存
4. 如果六项都达到 `31`，验证 `PERFECT_IVS` 成就
5. 然后继续沿着 `pokemonPrevolutions` 向前追溯，把前置形态也一并更新

因此 `DexEntry.ivs` 记录的不是“最近一次抓到的 IV”，而是“该物种线目前见过的每一项最高 IV”。

## 11. 图鉴默认展示属性的选取规则

`GameData.getSpeciesDefaultDexAttr(species, optimistic = false)` 用于从 `caughtAttr` 里选出一个默认展示组合。

### 11.1 非 optimistic 模式

1. 闪光位：
   - 优先 `NON_SHINY`
   - 如果没有 non-shiny 且有 shiny，则选 `SHINY`
2. 变体位：
   - 优先 `DEFAULT_VARIANT`
   - 否则 `VARIANT_2`
   - 否则 `VARIANT_3`
   - 都没有时仍回退 `DEFAULT_VARIANT`
3. 性别位：
   - 优先 `MALE`
   - 没有 male 且有 female 时才选 `FEMALE`
4. 形态位：
   - `getFormAttr(getFormIndex(attr))`

### 11.2 optimistic 模式

1. 如果已经解锁 shiny：
   - 默认展示 shiny
   - 并优先展示更高 variant
2. 否则展示 non-shiny 默认 variant

## 12. 图鉴与 starterData 的关系

图鉴本身存在 `dexData`，但 starter 相关解锁是分开存在 `starterData`：

- `moveset`
- `eggMoves`
- `candyCount`
- `friendship`
- `abilityAttr`
- `passiveAttr`
- `valueReduction`
- `classicWinCount`

其中和图鉴直接联动的有：

1. 抓到 starter 物种时解锁能力位
2. 孵蛋 / 捕获会推动 starter candy 统计
3. starter 选人界面会同时读取 `dexData` 与 `starterData`

## 13. 图鉴数据来自哪些事件

源码中图鉴更新主要来自以下事件：

1. 遭遇生成时：
   - `EncounterPhase.start()` 中对敌人调用 `setPokemonSeen()`
2. 宝可梦捕获成功时：
   - 调用 `setPokemonCaught()`
3. 孵蛋成功时：
   - `EggHatchData` 中调用 `setPokemonCaught(..., fromEgg = true)`
4. 某些 ME / 特殊奖励中获得宝可梦时：
   - 也会显式调用 `setPokemonCaught()` 和 `updateSpeciesDexIvs()`
5. 进化或其它变化后：
   - 若允许更新，也可能调用 `setPokemonSeen()` / `setPokemonCaught(false)`
